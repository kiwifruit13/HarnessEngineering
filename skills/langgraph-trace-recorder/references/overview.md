## 核心定位

本 Skill 的**最终目的**是：**将模型的执行过程转化为可观察、可分析、可复用的结构化知识，为模型的复杂推理提供上下文锚点，为复盘提供结构化依据。**

### 设计理念

- **白盒化**：将模型的"黑盒"执行过程转化为可观察的结构化数据
- **知识化**：将执行经验转化为可复用的知识资产
- **锚点化**：为模型推理提供历史上下文和经验参考

---

## 为什么需要这个 Skill？

### 问题的本质：模型的执行过程是"黑盒"

在传统的 LangGraph 应用中：

```
输入 → [黑盒执行] → 输出
```

**存在的痛点**：

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **不可观察** | 无法看到模型如何思考、如何决策 | 难以调试和优化 |
| **不可追溯** | 不知道之前类似的任务是如何执行的 | 无法复用成功经验 |
| **不可分析** | 失败后无法定位问题根源 | 难以改进决策路径 |
| **不可复用** | 执行经验随任务结束而丢失 | 每次都要重新试错 |

### 解决方案：将执行过程"白盒化"

```
输入 → [可观察的执行过程] → 输出
        ↓
   结构化知识图谱
   (问答轨迹 + 命令轨迹)
```

**实现方式**：

1. **捕获执行状态**：通过 LangGraph Callback 自动捕获节点执行状态
2. **分类执行行为**：自动识别问答（认知层）和命令（行动层）
3. **构建知识图谱**：将执行过程转化为节点-边关系的图谱
4. **提供结构化数据**：支持查询、分析和复用

## 双轨迹的深层含义

### 为什么要区分双轨迹？

LangGraph 应用中，模型的执行包含两个核心维度：

| 轨迹类型 | 作用 | 代表能力 | 示例 |
|---------|------|---------|------|
| **问答轨迹** | 记录思考过程 | 推理、决策、生成、理解 | "我需要先理解用户的需求..." |
| **命令轨迹** | 记录行动过程 | 工具调用、外部操作、数据获取 | "调用搜索工具查找信息..." |

### 完整链路

```
问题 → [问答推理] → [命令执行] → 结果 → [复盘分析]
  ↑                                              ↓
  └────────────── 历史经验复用 ──────────────────┘
```

### 双轨迹的价值

1. **分离关注点**：
   - 问答轨迹关注"为什么这样做"（推理）
   - 命令轨迹关注"做了什么"（行动）

2. **支持不同分析**：
   - 问答轨迹 → 分析决策逻辑
   - 命令轨迹 → 分析执行效率

3. **便于知识复用**：
   - 相似问题的推理路径可复用
   - 相似任务的执行步骤可复用

## 核心价值

### 1. 推理锚点（Reasoning Anchor）

为模型的复杂推理提供**上下文锚点**：

**核心能力**：

| 能力 | 说明 | 应用场景 |
|------|------|---------|
| **历史追溯** | 查询"之前类似的任务是如何执行的？" | 复用成功经验 |
| **因果分析** | 理解"为什么这个决策导致了那个结果？" | 优化决策路径 |
| **路径优化** | 识别"有没有更优的执行路径？" | 减少试错成本 |

**示例场景**：

```python
# 场景：模型需要分析2024年财报
query = "分析公司2024年的财务状况"

# → 查询知识图谱，找到类似的"分析2023年财报"任务
# → 学习执行路径：
#   1. 下载PDF财报
#   2. 解析财务数据
#   3. 计算关键指标
#   4. 生成分析报告
# → 应用：复用成功的执行路径，避免重复试错
```

**价值量化**：

- 减少试错时间：50-70%
- 提高成功率：30-50%
- 降低错误率：40-60%

### 2. 复盘基础（Review Foundation）

为任务复盘提供**结构化依据**：

**核心能力**：

| 能力 | 说明 | 应用场景 |
|------|------|---------|
| **成功复盘** | 识别关键成功因素 | 形成最佳实践 |
| **失败复盘** | 定位失败原因和恢复建议 | 避免重复错误 |
| **模式挖掘** | 发现"什么样的问答导致什么样的命令" | 优化执行策略 |

**示例场景 - 复盘报告**：

```markdown
## 任务执行复盘

### 执行概览
- 任务ID: task_1735689025
- 执行时间: 2024-01-01 10:00:00
- 节点总数: 15
- 成功节点: 12
- 失败节点: 3

### 成功节点分析
✓ query_handler - 成功解析用户意图（耗时：0.5s）
✓ download_pdf - 成功下载财报（耗时：2.5s）
✓ parse_data - 成功解析财务数据（耗时：1.2s）

### 失败节点分析
❌ fetch_api - 获取API数据失败（错误：ConnectionError）
   - 错误原因：网络连接超时
   - 恢复建议：增加重试机制，或使用备用数据源
   - 影响范围：导致后续计算步骤无法执行

### 优化建议
1. 在网络请求前增加连通性检查
2. 对失败的API调用实现自动重试（最多3次）
3. 提供备用数据源，确保任务可完成
```

**价值量化**：

- 复盘效率：提升 3-5 倍
- 问题定位：从"凭感觉"到"有数据支撑"
- 优化决策：从"猜测"到"基于证据"

### 3. 知识复用（Knowledge Reuse）

将执行经验转化为**可复用的知识资产**：

**核心能力**：

| 能力 | 说明 | 应用场景 |
|------|------|---------|
| **成功路径提取** | 提取成功的执行路径 | 形成执行模板 |
| **失败案例学习** | 分析失败原因 | 形成负面样本 |
| **错误模式识别** | 识别常见错误 | 形成防御策略 |

**示例场景 - 知识库构建**：

```python
# 场景：构建财务分析的知识库

# 成功案例
success_cases = [
    {
        "query": "分析2023年财报",
        "path": ["下载PDF", "解析数据", "计算指标", "生成报告"],
        "success_rate": 0.95
    }
]

# 失败案例
failure_cases = [
    {
        "query": "分析PDF财报",
        "failure_point": "解析数据",
        "error": "PDF格式不支持",
        "recovery": "先转换为标准格式"
    }
]

# 知识复用
def analyze_new_report(query):
    # 查询类似的成功路径
    similar = find_similar_query(query, success_cases)
    if similar:
        # 复用成功路径
        execute_path(similar["path"])
    else:
        # 检查可能的失败点
        check_failure_risks(query, failure_cases)
```

**价值量化**：

- 知识积累：每执行一次任务，知识库增长
- 复用率：30-50% 的任务可以复用历史经验
- 学习曲线：新任务上手时间缩短 40-60%

## 典型应用场景

### 场景 1：复杂推理任务

**场景描述**：模型需要处理复杂的多步推理任务

**使用方式**：
1. 记录推理过程的每一步
2. 构建推理路径图谱
3. 在新任务中复用成功路径

**收益**：
- 减少推理步骤：20-40%
- 提高推理质量：30-50%

### 场景 2：任务执行监控

**场景描述**：需要实时监控任务执行状态

**使用方式**：
1. 使用 Callback 自动捕获执行状态
2. 实时查看执行进度
3. 及时发现和解决问题

**收益**：
- 问题发现时间：从"事后"到"实时"
- 问题解决效率：提升 3-5 倍

### 场景 3：任务复盘优化

**场景描述**：任务完成后需要进行复盘分析

**使用方式**：
1. 导出结构化执行报告
2. 分析成功/失败节点
3. 生成优化建议

**收益**：
- 复盘效率：提升 3-5 倍
- 优化决策：从"猜测"到"基于证据"

### 场景 4：知识库构建

**场景描述**：需要积累和复用执行经验

**使用方式**：
1. 将执行过程转化为知识图谱
2. 存储到 Redis 实现持久化
3. 在新任务中查询和复用

**收益**：
- 知识积累：自动化
- 知识复用：30-50% 的任务可以复用

## 技术架构

### 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                     LangGraph 应用                        │
│                                                          │
│  ┌────────────┐     ┌────────────┐     ┌────────────┐  │
│  │  节点A     │ ──→ │  节点B     │ ──→ │  节点C     │  │
│  │ (问答)    │     │  (命令)    │     │  (问答)    │  │
│  └────────────┘     └────────────┘     └────────────┘  │
│        ↓                  ↓                  ↓         │
└────────┼──────────────────┼──────────────────┼─────────┘
         │                  │                  │
         └──────────────────┼──────────────────┘
                            ↓
                  ┌─────────────────┐
                  │  Callback      │
                  │  Handler       │
                  └────────┬────────┘
                           ↓
                  ┌─────────────────┐
                  │  Knowledge      │
                  │  Graph         │
                  │  Builder       │
                  └────────┬────────┘
                           ↓
                  ┌─────────────────┐
                  │  存储后端       │
                  │  - 内存存储     │
                  │  - Redis存储    │
                  └────────┬────────┘
                           ↓
                  ┌─────────────────┐
                  │  导出器         │
                  │  - JSON        │
                  │  - SVG         │
                  │  - Markdown    │
                  └─────────────────┘
```

### 核心组件

| 组件 | 文件 | 职责 |
|------|------|------|
| **Callback Handler** | `callback_handler.py` | 捕获 LangGraph 执行状态 |
| **Node Classifier** | `classifier.py` | 自动分类节点类型（问答/命令） |
| **Graph Builder** | `graph_builder.py` | 构建知识图谱 |
| **Storage Backend** | `storage_backend.py` | 存储后端抽象 |
| **Memory Storage** | `memory_storage.py` | 内存存储实现 |
| **Redis Storage** | `redis_storage.py` | Redis 存储实现 |
| **Graph Exporter** | `graph_exporter.py` | 多格式导出 |

### 数据流

```
1. LangGraph 执行节点
   ↓
2. Callback 捕获执行状态
   ↓
3. Classifier 分类节点类型
   ↓
4. Graph Builder 添加节点/边
   ↓
5. Storage Backend 存储数据
   ↓
6. Graph Exporter 导出结果
```
