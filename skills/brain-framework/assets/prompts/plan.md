# Plan 阶段 Prompt 模板

## 模板变量
- `{task}`: 任务描述
- `{research_summary}`: Research 阶段摘要
- `{reference_impl}`: 参考实现代码
- `{constraints}`: 约束条件
- `{output_path}`: 输出文件路径

## Prompt 内容

你是一个经验丰富的架构师。你的任务是基于研究结果，生成详细、可执行的实施计划。

## 任务目标
{task}

## Research 摘要
{research_summary}

## 参考实现
{reference_impl}

## 约束条件
{constraints}

## 计划要求

### 1. 清晰的目标
- 明确说明要实现什么功能
- 定义成功标准
- 列出验收条件

### 2. 完整的文件变更清单
列出所有需要修改的文件，包括：
- 文件路径
- 操作类型（CREATE/MODIFY/DELETE）
- 变更描述
- 依赖关系

### 3. 阶段划分
将实施过程划分为多个阶段：
- 每个阶段的目标
- 阶段的顺序
- 阶段间的依赖关系

### 4. 技术选型
- 选择的技术方案
- 选择原因
- 备选方案（如果适用）

### 5. 权衡决策
记录重要的技术决策：
- 决策点
- 可选方案
- 选定方案
- 决策原因

### 6. 风险评估
识别可能的风险：
- 技术风险
- 时间风险
- 资源风险

### 7. 回滚计划
提供回滚方案：
- 回滚条件
- 回滚步骤
- 回滚验证

## 输出格式

请生成一份详细的 `plan.md`，包含以下章节：

```markdown
# Implementation Plan - {task}

## 任务目标
{task 的详细说明}

## Research 摘要
{research_summary 的核心内容}

## 约束条件
- 约束1
- 约束2

## 实施方案

### 阶段划分
1. **阶段1**: 阶段1描述
2. **阶段2**: 阶段2描述
3. **阶段3**: 阶段3描述

### 文件变更清单
| 文件路径 | 操作类型 | 描述说明 | 依赖项 |
|---------|---------|---------|--------|
| file1.py | MODIFY | 修改描述 | file2.py |
| file2.py | CREATE | 创建描述 | - |

### 技术选型
- **技术点1**: 选择的技术和原因
- **技术点2**: 选择的技术和原因

### 权衡决策
| 决策点 | 可选方案 | 选定方案 | 原因说明 |
|-------|---------|---------|---------|
| 决策1 | 方案A, 方案B | 方案A | 原因说明 |

## 风险评估
- [ ] 风险1
- [ ] 风险2

## 回滚计划
1. 回滚步骤1
2. 回滚步骤2

## 审查批注区
（用户批注将追加到这里）
```

## 重要要求

### 防呆标记
在计划开头必须包含：
```markdown
> ⚠️ **当前状态**: 方案设计阶段，**尚未实施**
> 📝 **请批注**: 直接在本文件上进行修改和批注
> 🔄 **迭代**: 会根据批注更新方案
> ✅ **批准**: 审查完成后告诉我"批准，开始实施"
> 🚫 **重要**: don't implement yet
```

### 文件变更清单
- 必须使用表格格式
- 每个文件都必须描述清楚
- 明确标注依赖关系

### 权衡决策
- 记录所有重要的技术决策
- 说明选择某方案的原因
- 考虑备选方案

## 注意事项

1. **可执行性**: 计划必须具体、可执行，避免模糊描述
2. **完整性**: 不要遗漏任何需要修改的文件
3. **优先级**: 明确实施的优先级和顺序
4. **可验证性**: 每个阶段都应该有明确的验证标准
5. **回滚友好**: 确保每个阶段都可以独立回滚

## 输出文件
输出路径: `{output_path}`
格式: Markdown

## 示例

### 输入示例
```
task: "在用户模块添加邮箱验证功能"
research_summary: "用户模块包含 user.py 和 auth.py，使用 Flask 和 SQLAlchemy..."
constraints: "保持向后兼容，不破坏现有功能"
output_path: "plan.md"
```

### 输出示例片段
```markdown
# Implementation Plan - 在用户模块添加邮箱验证功能

## 任务目标
为用户模块添加邮箱验证功能，确保用户注册和更新邮箱时验证邮箱的有效性。

## 实施方案

### 阶段划分
1. **准备阶段**: 创建邮箱验证相关的数据库表
2. **核心实现**: 实现发送验证邮件和验证逻辑
3. **集成阶段**: 集成到注册和更新邮箱流程
4. **测试验证**: 单元测试和集成测试

### 文件变更清单
| 文件路径 | 操作类型 | 描述说明 | 依赖项 |
|---------|---------|---------|--------|
| models.py | MODIFY | 添加 EmailVerification 模型 | - |
| user.py | MODIFY | 添加发送验证邮件方法 | models.py |
| auth.py | MODIFY | 添加验证邮箱接口 | user.py |
| email_utils.py | CREATE | 邮件发送工具 | - |

### 技术选型
- **邮件服务**: 使用 SMTP，原因是简单可靠
- **验证码**: 6位数字，有效期30分钟

### 权衡决策
| 决策点 | 可选方案 | 选定方案 | 原因说明 |
|-------|---------|---------|---------|
| 邮件服务 | SMTP / 第三方API | SMTP | 无需额外依赖，足够安全 |
| 验证码长度 | 4位 / 6位 | 6位 | 安全性更好，用户可接受 |

...
```
