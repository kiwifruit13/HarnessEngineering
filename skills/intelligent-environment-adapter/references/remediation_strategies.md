# 补齐策略参考

## 概览

## 知识缺口补齐策略

### 1. search_external_knowledge

**适用场景**：
- 缺乏特定领域的知识
- 知识缺口可通过外部搜索补充
- 时效性要求不高

**实施步骤**：
1. 识别缺失的知识领域
2. 构建精准的搜索查询
3. 搜索外部知识源（文档、教程、API文档等）
4. 提取并整合关键知识
5. 更新知识库

**预期效果**：
- 知识覆盖率提升：+20-40%
- 响应时间：中等（30-60秒）

**成本估算**：
- 时间：30-60秒
- Tokens：1000-3000
- 计算：低

**适用层级**：L1, L2

---

### 2. build_domain_model

**适用场景**：
- 领域知识复杂且系统性强
- 需要构建结构化知识体系
- 任务需要深度理解领域概念

**实施步骤**：
1. 收集领域核心概念
2. 构建知识图谱或概念模型
3. 定义概念间的关系
4. 添加实例和属性
5. 验证模型完整性

**预期效果**：
- 知识覆盖率提升：+40-60%
- 响应时间：长（60-120秒）

**成本估算**：
- 时间：60-120秒
- Tokens：3000-8000
- 计算：中等

**适用层级**：L1, L2

---

### 3. fetch_expert_documents

**适用场景**：
- 需要专家级知识
- 有权威的专家文档可用
- 任务需要高准确性

**实施步骤**：
1. 识别专家文档来源（官方文档、学术论文、行业标准等）
2. 获取并解析文档
3. 提取关键知识点
4. 整合到知识库
5. 验证知识准确性

**预期效果**：
- 知识覆盖率提升：+30-50%
- 响应时间：中等（60-90秒）

**成本估算**：
- 时间：60-90秒
- Tokens：2000-5000
- 计算：低

**适用层级**：L1, L2

---

## 能力缺口补齐策略

### 4. search_and_load_skill

**适用场景**：
- 缺乏特定能力
- 有现成的skill可用
- 时效性要求高

**实施步骤**：
1. 识别缺失的能力
2. 搜索可用的skills
3. 评估候选skills的匹配度
4. 选择最优skill
5. 请求用户审批
6. 加载并集成skill

**预期效果**：
- 能力覆盖率提升：+50-80%
- 响应时间：中等（30-60秒）

**成本估算**：
- 时间：30-60秒
- Tokens：1000-2000（搜索） + skill本身的消耗
- 计算：低

**适用层级**：L3, L4

---

### 5. compose_multiple_skills

**适用场景**：
- 需要多个技能协同
- 单个skill无法满足需求
- 任务复杂度高

**实施步骤**：
1. 分解任务为子任务
2. 识别每个子任务需要的skill
3. 搜索并评估候选skills
4. 设计skills的组合方式
5. 定义执行顺序和数据流
6. 请求用户审批
7. 执行组合流程

**预期效果**：
- 能力覆盖率提升：+60-90%
- 响应时间：长（60-120秒）

**成本估算**：
- 时间：60-120秒
- Tokens：2000-5000 + 各skill的消耗
- 计算：中等

**适用层级**：L3, L4

---

### 6. delegate_to_human

**适用场景**：
- 能力无法通过skills补齐
- 需要人类的专业判断
- 任务涉及伦理或复杂决策

**实施步骤**：
1. 识别无法补齐的能力
2. 明确需要人类介入的环节
3. 设计人机协作流程
4. 向人类请求指导或决策
5. 整合人类的输入
6. 继续执行任务

**预期效果**：
- 能力覆盖率提升：依赖人类专业度
- 响应时间：不可控（等待人类响应）

**成本估算**：
- 时间：不可控
- Tokens：500-1000
- 计算：低

**适用层级**：L2, L4, L5

## 数据源缺口补齐策略

### 7. search_alternative_sources

**适用场景**：
- 原数据源不可用
- 需要替代数据源
- 数据类型可替换

**实施步骤**：
1. 识别缺失的数据源
2. 搜索替代数据源
3. 评估替代源的质量和可用性
4. 测试数据访问
5. 整合到工作流

**预期效果**：
- 数据可用性提升：+30-50%
- 响应时间：中等（30-60秒）

**成本估算**：
- 时间：30-60秒
- Tokens：1000-2000
- 计算：低

**适用层级**：L1

---

### 8. integrate_realtime_api

**适用场景**：
- 需要实时数据
- 有可用的实时API
- 数据更新频繁

**实施步骤**：
1. 识别实时数据需求
2. 搜索可用的实时API
3. 评估API的性能和稳定性
4. 集成API到工作流
5. 实现数据缓存和错误处理
6. 测试实时性

**预期效果**：
- 实时性提升：+80-100%
- 响应时间：中等（30-60秒）

**成本估算**：
- 时间：30-60秒
- Tokens：1000-2000
- 计算：低

**适用层级**：L1

---

### 9. use_cached_data

**适用场景**：
- 实时性要求不高
- 有可用的缓存数据
- 成本敏感

**实施步骤**：
1. 检查缓存中的数据
2. 评估数据的新鲜度
3. 如果数据有效，直接使用
4. 如果数据过期，考虑更新或使用替代源

**预期效果**：
- 成本降低：-50-70%
- 响应时间：极短（<10秒）

**成本估算**：
- 时间：<10秒
- Tokens：<100
- 计算：极低

**适用层级**：L1

## 处理缺口补齐策略

### 10. optimize_processing_pipeline

**适用场景**：
- 处理流程效率低
- 有优化空间
- 任务规模大

**实施步骤**：
1. 分析当前处理流程
2. 识别瓶颈和低效环节
3. 设计优化方案
4. 实施优化（并行化、缓存、批处理等）
5. 测试和验证优化效果

**预期效果**：
- 处理效率提升：+30-50%
- 响应时间：中等（30-60秒）

**成本估算**：
- 时间：30-60秒
- Tokens：1000-2000
- 计算：低

**适用层级**：L2

---

### 11. distribute_processing_tasks

**适用场景**：
- 任务规模大
- 可分解为独立子任务
- 有并行处理能力

**实施步骤**：
1. 分解任务为可并行的子任务
2. 设计任务分发机制
3. 配置并行处理资源
4. 监控任务执行状态
5. 汇总和处理结果

**预期效果**：
- 处理效率提升：+50-80%
- 响应时间：中等（30-60秒）

**成本估算**：
- 时间：30-60秒
- Tokens：1500-2500
- 计算：中等

**适用层级**：L2, L4

---

### 12. simplify_processing_requirements

**适用场景**：
- 处理需求过于复杂
- 可以简化而不影响核心目标
- 成本或时间受限

**实施步骤**：
1. 分析处理需求的必要性
2. 识别可简化的环节
3. 设计简化方案
4. 评估简化对结果的影响
5. 实施简化方案

**预期效果**：
- 复杂度降低：-30-50%
- 响应时间：中等（30-60秒）

**成本估算**：
- 时间：30-60秒
- Tokens：1000-2000
- 计算：低

**适用层级**：L2

## 策略选择规则

### 决策树

```
短板类型是 knowledge_gap？
├─ 是 → 时效性要求高？
│       ├─ 是 → search_external_knowledge
│       └─ 否 → 领域复杂？
│               ├─ 是 → build_domain_model
│               └─ 否 → fetch_expert_documents
└─ 否 → 短板类型是 capability_gap？
        ├─ 是 → 有现成skill？
        │       ├─ 是 → search_and_load_skill
        │       └─ 否 → 需要多技能协同？
        │               ├─ 是 → compose_multiple_skills
        │               └─ 否 → delegate_to_human
        └─ 否 → 短板类型是 data_source_gap？
                ├─ 是 → 时效性要求高？
                │       ├─ 是 → integrate_realtime_api
                │       └─ 否 → search_alternative_sources
                └─ 否 → 短板类型是 processing_gap？
                        ├─ 是 → 任务规模大？
                        │       ├─ 是 → distribute_processing_tasks
                        │       └─ 否 → optimize_processing_pipeline
                        └─ 否 → 短板类型是 integration_gap？
                                └─ build_adapter_layer
```

### 优先级计算

```
策略优先级 = (短板严重性 × 0.4) + (预期提升效果 × 0.3) + (成本可控性 × 0.2) + (实施难度 × 0.1)
```

其中：
- 短板严重性：critical=1.0, moderate=0.7, minor=0.4
- 预期提升效果：0.0-1.0（基于历史数据）
- 成本可控性：0.0-1.0（成本越低得分越高）
- 实施难度：0.0-1.0（难度越低得分越高）
