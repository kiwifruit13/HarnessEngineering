## 标准决策案例

### 案例1：简单查询任务（MODE1）

**用户输入**：
"帮我查一下Python中list的sort方法怎么用"

**任务特征评估**：
```
□ 紧急度：low
□ 复杂度：0.2（简单操作）
□ 优先级：3（常规查询）
```

**决策过程**：
```
紧急度=critical？ 否 ↓
复杂度<0.3？ 是 → MODE1
```

**执行方式**：
```
Python的list.sort()方法用于原地排序列表。

基本用法：
my_list = [3, 1, 2]
my_list.sort()  # 结果：[1, 2, 3]

常用参数：
- reverse=True：降序排序
- key=函数：自定义排序规则

示例：
my_list.sort(reverse=True)  # 降序：[3, 2, 1]
```

### 案例2：问题分析任务（MODE2）

**用户输入**：
"我的程序运行很慢，帮我分析可能的原因"

**任务特征评估**：
```
□ 紧急度：medium
□ 复杂度：0.5（需要分析多个因素）
□ 优先级：6（影响用户体验）
□ 任务描述>50字 → 复杂度+0.2
```

**决策过程**：
```
紧急度=critical？ 否 ↓
复杂度>0.7 且 优先级≥8？ 否 ↓
复杂度<0.3？ 否 ↓
→ MODE2（默认）
```

**执行方式**：
```
Step 1: 算法复杂度分析
检查是否有O(n²)或更高复杂度的算法
使用性能分析工具定位热点代码

Step 2: 数据量评估
确认处理的数据量是否超出预期
评估数据结构是否合适

Step 3: I/O瓶颈排查
检查是否有频繁的磁盘读写
查看是否有网络请求阻塞

Step 4: 内存问题检查
监控内存使用情况
检查是否有内存泄漏

Step 5: 并发问题分析
分析是否存在线程竞争
检查锁等待情况

建议：先用性能分析工具（如cProfile）定位具体瓶颈，再针对性优化。
```

### 案例3：重大决策任务（MODE3）

**用户输入**：
"公司要选择技术架构方案，关系到未来3年发展。方案A：微服务；方案B：单体；方案C：混合架构。"

**任务特征评估**：
```
□ 紧急度：medium
□ 复杂度：0.9（涉及多个因素，长期影响）
□ 优先级：9（重要决策）
□ 包含"设计"、"架构" → 复杂度+0.3
□ 用户强调"重要"、"关系到未来" → 优先级+2
```

**决策过程**：
```
紧急度=critical？ 否 ↓
复杂度>0.7 且 优先级≥8？ 是 → MODE3
```

**执行方式**：
```
【方案A：微服务架构】

优势：
- 技术灵活性：不同服务可使用最适合的技术栈
- 独立部署：单个服务更新不影响其他服务
- 可扩展性：可针对瓶颈服务单独扩容
- 容错性：单服务故障不影响整体系统

挑战：
- 运维复杂度高：需要服务发现、配置中心等基础设施
- 分布式事务处理复杂
- 服务间通信开销

适用场景：
- 团队规模较大，需要独立开发部署
- 业务边界清晰，可拆分为独立服务
- 对可扩展性要求高

【方案B：单体架构】

优势：
- 开发简单：一个项目，技术栈统一
- 部署方便：一次部署，无需复杂的编排
- 性能好：无服务间通信开销
- 调试容易：问题定位直接

挑战：
- 技术栈受限：所有功能使用同一技术栈
- 扩展性受限：只能整体扩容
- 耦合度高：代码改动影响面大

适用场景：
- 团队规模小
- 项目初期，业务模式不清晰
- 对快速迭代要求高

【方案C：混合架构】

优势：
- 核心服务微服务化：关键模块独立部署
- 非核心服务单体化：减少运维复杂度
- 渐进式演进：可以先单体后微服务

挑战：
- 架构复杂度中等
- 需要明确哪些服务独立

适用场景：
- 过渡期项目
- 部分模块需要独立扩展

【决策建议】

基于未来3年发展考虑：

1. 如果团队<10人，业务快速迭代期
   → 推荐：单体架构（方案B）
   理由：快速上线，避免过度设计

2. 如果团队10-30人，业务相对成熟
   → 推荐：混合架构（方案C）
   理由：核心服务独立，非核心保持简单

3. 如果团队>30人，业务边界清晰
   → 推荐：微服务架构（方案A）
   理由：支持团队独立开发，业务解耦

【元认知反思】
架构选择没有绝对对错，关键是匹配团队规模和业务阶段。避免"微服务崇拜"，也不要因为简单而拒绝演进。建议每半年重新评估架构合理性，保持架构与业务的动态匹配。
```

```

□ 复杂度：0.4（有标准流程）
□ 优先级：10
```

**决策过程**：
```
紧急度=critical？ 是 → MODE0
```

**执行方式**：
```
1. 检查服务状态：systemctl status service_name

3. 重启服务：systemctl restart service_name
4. 如无法启动，检查配置文件和依赖服务
```

## 边界情况案例

### 案例5：复杂度边界（MODE1 vs MODE2）

**用户输入**：
"帮我优化这段代码的性能"

**评估挑战**：
- 复杂度判断：可能是简单优化（MODE1），也可能需要深度分析（MODE2）

**决策方法**：
```
第一步：快速判断复杂度
- 如果代码<50行 → 复杂度设为0.3 → MODE1
- 如果代码>50行 → 复杂度设为0.5 → MODE2

第二步：动态调整
- 开始用MODE1，发现需要深度分析 → 升级到MODE2
- 开始用MODE2，发现可以快速解决 → 降级到MODE1
```

**推荐策略**：
先用MODE1快速评估，如发现复杂，立即升级到MODE2。

## 总结

**模式选择的核心原则**：

1. **快速判断**：30秒内完成评估，避免过度分析
2. **保守选择**：不确定时优先选择MODE2（默认模式）
3. **动态调整**：执行中发现不匹配，立即调整
4. **持续优化**：记录案例，优化决策规则

**常见误区**：

- ❌ 过度思考：简单任务使用MODE3 → 浪费资源
- ❌ 思考不足：复杂任务使用MODE0 → 质量下降
- ❌ 忽略紧急度：critical任务仍用MODE2 → 延误时机
- ❌ 僵化执行：不根据实际情况调整 → 适用性差
