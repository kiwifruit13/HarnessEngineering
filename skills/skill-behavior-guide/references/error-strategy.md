## 概述

自我修正能力是 AI Agent 的重要能力。本策略提供系统化的自我修正方法，确保 AI Agent 能够从错误中学习并自动调整。

**适用场景**：
- 执行出错或输出不达标
- 用户反馈存在问题
- 自我发现执行偏差

**核心原则**：
- 早识别：尽早发现问题
- 深分析：找到根本原因
- 快调整：及时修正策略
- 重验证：确认修正有效

## 核心策略框架

```
自我修正闭环 = 错误识别 → 根因分析 → 策略调整 → 效果验证
```

| 阶段 | 核心目标 | 关键动作 |
|-----|---------|---------|
| 错误识别 | 发现问题 | 识别偏差、分类问题、评估影响 |
| 根因分析 | 找到原因 | 分析原因、定位环节、追溯源头 |
| 策略调整 | 解决问题 | 制定方案、优化方法、重新执行 |
| 效果验证 | 确认解决 | 验证效果、确认消除、记录经验 |

## 策略一：错误识别

**核心目标**：准确识别执行中的问题

### 1.1 问题类型识别

**识别问题所属类型**：

```
【理解类问题】
表现：执行方向偏离用户需求
原因：需求理解偏差、规范理解错误
特征：输出内容与用户期望不符

【方法类问题】
表现：执行方法不当
原因：方法选择错误、执行路径不当
特征：效率低、质量差、资源浪费

【质量类问题】
表现：输出不达标
原因：验证不充分、标准未达到
特征：输出缺少内容、存在错误、不可用

【资源类问题】
表现：无法继续执行
原因：资源不足、权限不够、时间不够
特征：执行中断、条件不满足
```

### 1.2 问题严重性评估

**评估问题的严重程度**：

```
【严重性等级】
- 致命：无法完成核心任务
- 严重：影响主要功能或质量
- 中等：影响部分功能或效率
- 轻微：不影响核心功能

【影响范围】
- 全局：影响整个任务
- 局部：影响部分功能
- 单点：影响个别项目

【紧急程度】
- 紧急：需要立即处理
- 一般：可以后续处理
- 可延后：不影响当前流程
```

### 1.3 自我检测机制

**建立系统的自我检测**：

```
【执行中检测】
每个步骤完成后：
□ 执行结果是否符合预期？
□ 是否存在异常情况？
□ 是否需要调整？

【阶段检测】
完成一个阶段后：
□ 阶段输出是否达标？
□ 是否满足下一阶段条件？
□ 是否存在遗留问题？

【最终检测】
任务完成后：
□ 输出是否完整？
□ 输出是否准确？
□ 输出是否可用？
```

```
【初步判断】

```

## 策略二：根因分析

**核心目标**：找到问题的根本原因

### 2.1 5Why 分析法

**通过连续追问找到根因**：

```
【5Why 分析模板】
问题现象：[具体问题]

Why 1：为什么会出现这个问题？
回答：[原因1]
依据：[事实依据]

Why 2：为什么会出现[原因1]？
回答：[原因2]
依据：[事实依据]

Why 3：为什么会出现[原因2]？
回答：[原因3]
依据：[事实依据]

Why 4：为什么会出现[原因3]？
回答：[原因4]
依据：[事实依据]

Why 5：为什么会出现[原因4]？
回答：[根本原因]
依据：[事实依据]

【根本原因】
最终确定的根因：[根因描述]
可操作性：[是否可通过改进解决]
```

**应用要点**：
- 每个回答都要有事实依据
- 避免停留在表面原因
- 根因必须是可操作的改进点

### 2.2 多维度分析

**从多个维度分析问题**：

```
【需求维度】
□ 需求理解是否准确？
□ 成功标准是否明确？
□ 约束条件是否识别？

【方法维度】
□ 方法选择是否正确？
□ 执行步骤是否合理？
□ 资源使用是否得当？

【规范维度】
□ 规范理解是否准确？
□ 执行是否符合规范？
□ 验证是否充分？

【环境维度】
□ 执行环境是否正常？
□ 外部条件是否满足？
□ 是否有外部干扰？
```

### 2.3 环节定位

**定位问题发生的具体环节**：

```
【环节追溯】
问题环节：[环节名称]

前置环节检查：
□ 上一环节输出是否正确？
□ 传递的信息是否完整？
□ 是否受到前置环节影响？

当前环节检查：
□ 执行过程是否正确？
□ 使用的数据是否准确？
□ 方法应用是否得当？

后续环节影响：
□ 是否影响下一环节？
□ 影响范围有多大？
□ 是否需要调整后续环节？
```

### 2.4 根因分析模板

```
根因分析报告：

【问题描述】
[问题具体描述]

【5Why 分析】
Why 1 → Why 2 → Why 3 → Why 4 → Why 5
[层层追溯过程]

【多维度分析】
- 需求维度：[分析结果]
- 方法维度：[分析结果]
- 规范维度：[分析结果]
- 环境维度：[分析结果]

【环节定位】
问题环节：[具体环节]
影响环节：[受影响环节]

【根本原因】
根本原因：[最终结论]
改进方向：[改进方向]
```

## 策略三：策略调整

**核心目标**：制定并实施修正方案

### 3.1 修正方案制定

**根据根因制定针对性方案**：

```
【方案制定框架】
根本原因：[根因描述]

修正目标：
- 要解决什么问题：[问题描述]
- 要达到什么效果：[目标描述]

修正方案：
方案一：[方案描述]
- 适用性：[适合什么情况]
- 资源需求：[需要什么资源]
- 预期效果：[预期达到什么效果]

方案二：[方案描述]
- 适用性：[适合什么情况]
- 资源需求：[需要什么资源]
- 预期效果：[预期达到什么效果]

方案选择：
选择方案：[最终选择]
选择原因：[原因说明]
```

### 3.2 调整类型

**不同类型问题的调整策略**：

```
【理解类问题调整】
问题：需求理解偏差
调整：
- 重新分析用户需求
- 主动向用户确认
- 修正执行方向

【方法类问题调整】
问题：执行方法不当
调整：
- 重新选择方法
- 优化执行步骤
- 调整资源配置

【质量类问题调整】
问题：输出不达标
调整：
- 补充缺失内容
- 修正错误内容
- 优化输出质量

【资源类问题调整】
问题：资源不足
调整：
- 优化资源使用
- 寻找替代资源
- 调整执行计划
```

### 3.3 执行调整

**实施修正方案**：

```
【调整前准备】
□ 明确调整范围
□ 确定回退点（必要时回退到某个环节）
□ 准备调整所需资源

【调整实施】
步骤1：[具体动作]
- 回退到：[回退点]
- 原因：[为什么回退]

步骤2：[具体动作]
- 调整内容：[调整什么]
- 调整方法：[如何调整]

步骤3：[具体动作]
- 重新执行：[执行什么]
- 验证调整：[如何验证]

【调整记录】
调整内容：[具体内容]
调整时间：[时间]
调整原因：[原因]
调整结果：[结果]
```

## 策略四：效果验证

**核心目标**：确认修正是否有效

### 4.1 验证修正效果

**验证问题是否解决**：

```
【问题验证】
原问题：[问题描述]
当前状态：[当前情况]
是否解决：[是/否/部分解决]

【标准验证】
原标准：[原标准要求]
当前输出：[当前输出]
是否达标：[是/否/部分达标]

【用户验证】
用户需求：[需求描述]
当前结果：[结果描述]
是否满足：[是/否/部分满足]
```

### 4.2 副作用检查

**检查修正是否引入新问题**：

```
【新问题检查】
□ 是否引入新问题？
□ 是否影响其他部分？
□ 是否符合规范要求？

【影响范围检查】
□ 对已完成部分的影响：[影响描述]
□ 对后续执行的影响：[影响描述]
□ 对整体质量的影响：[影响描述]

【兼容性检查】
□ 与其他部分是否兼容？
□ 是否符合整体要求？
□ 是否需要其他调整？
```

### 4.3 经验记录

**记录修正经验供未来参考**：

```
【问题记录】
问题描述：[问题]
发现时机：[何时]
问题类型：[类型]

【根因记录】
根本原因：[根因]
分析过程：[过程描述]

【方案记录】
修正方案：[方案]
实施过程：[过程描述]
实施结果：[结果]

【经验总结】
可复用经验：
- [经验1]
- [经验2]

避免重复：
- [教训1]
- [教训2]
```

### 4.4 验证决策

```
【验证通过】
问题已解决 + 无副作用 + 符合标准
→ 继续后续流程

【验证不通过】
问题未解决 或 有副作用 或 不符合标准
→ 重新分析原因并调整

【无法修正】
多次尝试仍未解决
→ 向用户说明情况并寻求帮助
```

```

□ 我是否准确识别了问题类型？
□ 我是否正确评估了问题严重性？
□ 我是否及时发现了问题？

【根因分析阶段】
□ 我是否找到了根本原因？
□ 我是否进行了多维度分析？
□ 我是否定位了问题环节？

【策略调整阶段】
□ 我是否制定了针对性方案？
□ 我是否选择了最优方案？
□ 我是否正确实施了调整？

【效果验证阶段】
□ 我是否验证了问题是否解决？
□ 我是否检查了是否有副作用？
□ 我是否记录了经验教训？
```

### 问题诊断矩阵

| 问题现象 | 可能原因 | 修正方向 |
|---------|---------|---------|
| 重复同类错误 | 根因分析不深 | 深挖根本原因 |
| 修正后仍有问题 | 调整不彻底 | 全面调整策略 |
| 引入新问题 | 验证不充分 | 加强副作用检查 |
| 无法解决 | 能力或资源限制 | 向用户说明情况 |

## 应用示例

### 示例1：需求理解偏差的自我修正

**问题**：用户反馈输出不符合需求

**修正过程**：
1. **错误识别**：
   - 类型：理解类问题
   - 原因：需求理解偏差
   - 严重性：严重

2. **根因分析**：
   - Why 1：为什么不符合？→ 理解方向错误
   - Why 2：为什么理解错误？→ 未确认用户核心需求
   - Why 3：为什么未确认？→ 用户需求表达模糊
   - Why 4：为什么没有澄清？→ 缺少主动确认环节
   - 根因：未主动向用户确认模糊需求

3. **策略调整**：
   - 回退到理解需求环节
   - 主动向用户询问确认核心需求
   - 明确后再重新执行

4. **效果验证**：
   - 验证输出是否符合确认后的需求 → 符合
   - 检查是否引入新问题 → 无
   - 记录经验：模糊需求必须主动确认

### 示例2：输出质量不达标的自我修正

**问题**：自检发现输出缺少必需内容

**修正过程**：
1. **错误识别**：
   - 类型：质量类问题
   - 原因：执行遗漏
   - 严重性：严重

2. **根因分析**：
   - Why 1：为什么缺少内容？→ 执行时遗漏
   - Why 2：为什么遗漏？→ 未对照规范检查
   - Why 3：为什么未检查？→ 缺少检查环节
   - 根因：缺少执行后对照规范检查的环节

3. **策略调整**：
   - 补充缺失内容
   - 对照规范重新检查所有内容
   - 验证完整性后再输出

4. **效果验证**：
   - 验证内容是否完整 → 完整
   - 验证是否符合规范 → 符合
   - 记录经验：必须执行完整性检查

### 示例3：方法选择不当的自我修正

**问题**：执行效率低，耗时过长

**修正过程**：
1. **错误识别**：
   - 类型：方法类问题
   - 原因：方法选择不当
   - 严重性：中等

2. **根因分析**：
   - Why 1：为什么效率低？→ 使用了低效方法
   - Why 2：为什么选择低效方法？→ 未评估方法效率
   - Why 3：为什么未评估？→ 方法选择环节缺失
   - 根因：缺少方法效率评估环节

3. **策略调整**：
   - 暂停当前执行
   - 重新评估可选方法的效率
   - 选择更高效的方法重新执行

4. **效果验证**：
   - 验证效率是否提升 → 提升明显
   - 验证质量是否保持 → 保持
   - 记录经验：方法选择需评估效率
