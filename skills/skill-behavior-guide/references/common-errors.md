## 概述

本案例库收集了 AI Agent 执行 Skill 时常见的遵循失败案例，旨在通过反面教材提高遵循意识和能力。

**使用方法**：
- 执行前阅读，建立风险意识
- 遇到类似场景时，对照规避
- 自我修正时，参考根因分析

## 错误分类体系

| 错误类型 | 发生阶段 | 典型表现 | 影响程度 |
|----------|----------|----------|----------|
| 规范理解不完整 | 执行前 | 遗漏关键要求 | 致命 |
| 必选步骤遗漏 | 执行中 | 跳过必需步骤 | 严重 |
| 禁止事项违反 | 执行中 | 触犯规范红线 | 致命 |
| 输出标准不达标 | 执行后 | 输出缺少必需内容 | 严重 |
| 前置条件忽略 | 执行前 | 不满足条件强行执行 | 严重 |
| 执行中对照缺失 | 执行中 | 不对照规范执行 | 中等 |
| 验证环节敷衍 | 执行后 | 验证不充分就输出 | 中等 |

## 案例一：规范理解不完整

### 场景描述

Agent 执行一个数据处理 Skill，规范要求：
- 必选步骤：数据验证 → 数据清洗 → 数据转换 → 数据输出
- 禁止事项：不得修改原始数据
- 输出标准：必须包含数据质量报告

Agent 只关注了必选步骤，遗漏了"禁止事项"和"输出标准"。

### 错误表现

```
执行过程：
1. 数据验证 ✓
2. 数据清洗 ✓
3. 数据转换 ✓
4. 数据输出 ✓

遗漏：
- 未读取"禁止事项"，在清洗时修改了原始数据
- 未读取"输出标准"，未生成数据质量报告
```

### 根因分析

```
Why 1：为什么遗漏了关键要求？
→ 只阅读了必选步骤，未完整阅读规范

Why 2：为什么未完整阅读？
→ 认为必选步骤是核心，其他是次要

Why 3：为什么有这种认知？
→ 缺少"规范完整性检查"环节

根因：缺少规范阅读完整性验证机制
```

### 规避方法

```
✅ 规范阅读清单：
□ 必选步骤
□ 可选步骤
□ 禁止事项
□ 输出标准
□ 前置条件

✅ 阅读后自检：
"我能否说出所有禁止事项？"
"我能否说出输出必须包含什么？"
如不能回答，重新阅读。
```

## 案例二：必选步骤遗漏

### 场景描述

Agent 执行一个报告生成 Skill，规范要求：
- 必选步骤：数据收集 → 数据分析 → 可视化生成 → 报告撰写 → 质量检查

Agent 跳过了"质量检查"步骤，直接输出报告。

### 错误表现

```
执行过程：
1. 数据收集 ✓
2. 数据分析 ✓
3. 可视化生成 ✓
4. 报告撰写 ✓
5. [跳过] 质量检查 ✗

输出：报告存在数据错误
```

### 根因分析

```
Why 1：为什么跳过质量检查？
→ 认为报告撰写完成即任务完成

Why 2：为什么有这种认知？
→ 未明确区分"可选步骤"和"必选步骤"

Why 3：为什么未区分？
→ 规范阅读时未标记步骤属性

根因：未建立步骤属性识别机制
```

### 规避方法

```
✅ 步骤标记法：
必选步骤：标记 [MUST]，不可跳过
可选步骤：标记 [OPT]，按需执行

✅ 执行清单：
每完成一步，在清单中勾选
执行完毕后，确认所有 [MUST] 已勾选

✅ 遗漏检测：
"是否还有未执行的必选步骤？"
```

## 案例三：禁止事项违反

### 场景描述

Agent 执行一个配置修改 Skill，规范明确禁止：
- 禁止事项：不得删除原有配置项

Agent 在添加新配置时，删除了原有配置中"看似无用"的项。

### 错误表现

```
执行过程：
1. 读取配置文件
2. 添加新配置项
3. [违反] 删除了"注释掉的配置项"
4. 输出修改后的配置

后果：被删除的配置项实际是备份配置，系统回滚失败
```

### 根因分析

```
Why 1：为什么删除了配置项？
→ 认为注释掉的项是无用数据

Why 2：为什么没意识到违反禁止事项？
→ 执行前未明确识别禁止事项

Why 3：为什么未识别？
→ 规范阅读时跳过了"禁止事项"部分

根因：对禁止事项重视不足，缺少显式识别环节
```

### 规避方法

```
✅ 禁止事项显式列表：
执行前，列出所有禁止事项：
[禁止1] 不得删除原有配置项
[禁止2] 不得修改只读字段
[禁止3] ...

✅ 执行前确认：
"我要执行的每个操作，是否触犯禁止事项？"

✅ 触犯检测：
任何删除/修改操作前，检查是否在禁止列表中
```

## 案例四：输出标准不达标

### 场景描述

Agent 执行一个数据分析 Skill，规范要求输出必须包含：
- 数据概览
- 趋势分析
- 异常检测
- 行动建议

Agent 输出缺少"异常检测"部分。

### 错误表现

```
输出内容：
1. 数据概览 ✓
2. 趋势分析 ✓
3. [缺失] 异常检测 ✗
4. 行动建议 ✓

验证：未逐项对照规范验证
```

### 根因分析

```
Why 1：为什么缺少异常检测？
→ 执行时遗漏了该部分

Why 2：为什么没有发现遗漏？
→ 输出前未对照规范逐项验证

Why 3：为什么未验证？
→ 认为输出内容完整，凭感觉判断

根因：缺少输出完整性验证清单
```

### 规避方法

```
✅ 输出清单：
规范要求输出项：
□ 数据概览
□ 趋势分析
□ 异常检测
□ 行动建议

输出前逐项勾选，全部勾选才输出

✅ 验证模板：
"我的输出是否包含规范要求的所有内容？"
逐项对照，确认每项都存在
```

## 案例五：前置条件忽略

### 场景描述

Agent 执行一个数据库操作 Skill，规范要求前置条件：
- 数据库连接已建立
- 表结构已确认
- 备份已完成

Agent 在数据库连接未稳定时就开始执行。

### 错误表现

```
执行前检查：
- 数据库连接：[不稳定，忽略]
- 表结构确认：[未执行]
- 备份完成：[未执行]

执行结果：操作中途连接断开，数据损坏
```

### 根因分析

```
Why 1：为什么忽略前置条件？
→ 认为条件"差不多满足"，想尽快执行

Why 2：为什么会有这种判断？
→ 未逐项验证前置条件是否完全满足

Why 3：为什么未逐项验证？
→ 前置条件检查流于形式

根因：前置条件检查不严格，存在侥幸心理
```

### 规避方法

```
✅ 前置条件逐项验证：
□ 条件1：数据库连接已建立 [是/否]
□ 条件2：表结构已确认 [是/否]
□ 条件3：备份已完成 [是/否]

全部为"是"才开始执行

✅ 不满足处理：
任一条件不满足 → 不执行，向用户说明缺失项
```

## 案例六：执行中对照缺失

### 场景描述

Agent 执行一个复杂 Skill，包含 15 个步骤。执行过程中未对照规范，导致执行偏离。

### 错误表现

```
步骤执行：
步骤1-5：按规范执行 ✓
步骤6-10：凭记忆执行，偏离规范 ✗
步骤11-15：继续偏离 ✗

验证时才发现多处不符合规范要求
```

### 根因分析

```
Why 1：为什么执行偏离？
→ 执行中未对照规范，凭记忆执行

Why 2：为什么未对照？
→ 步骤多，认为对照会降低效率

Why 3：为什么有这种认知？
→ 未意识到偏离的代价远高于对照的成本

根因：低估偏离风险，高估对照成本
```

### 规避方法

```
✅ 关键节点对照：
每完成 3-5 个步骤，对照规范确认：
- 执行方法是否正确？
- 输出是否符合预期？
- 是否有偏离迹象？

✅ 阶段验证：
完成一个阶段后，暂停，验证阶段输出
确认符合规范后再进入下一阶段
```

## 案例七：验证环节敷衍

### 场景描述

Agent 执行完成后进行验证，但验证流于形式，未发现问题。

### 错误表现

```
验证过程：
□ 输出完整性：草草浏览，认为完整 ✓（实际遗漏）
□ 输出质量：凭感觉判断，认为达标 ✓（实际有错）
□ 规范符合性：未逐项对照，认为符合 ✓（实际不符）

输出：不达标的输出被提交
```

### 根因分析

```
Why 1：为什么验证敷衍？
→ 急于完成任务，心态浮躁

Why 2：为什么心态浮躁？
→ 认为验证是形式，不是核心

Why 3：为什么有这种认知？
→ 未认识到验证是遵循的最后一道防线

根因：对验证的重要性认知不足
```

### 规避方法

```
✅ 验证清单逐项执行：
不凭感觉，逐项对照：
□ 内容A是否存在？[检查具体位置]
□ 内容B是否正确？[验证具体数据]
□ 格式C是否符合？[对照具体标准]

✅ 验证深度：
不满足于"看起来没问题"
要确认"每一项都符合规范"

✅ 验证态度：
验证是最后的防线，敷衍等于放弃遵循
```

## 规避方法总结

### 遵循七戒

```
一戒：规范阅读不完整
   → 必读全：必选步骤、禁止事项、输出标准、前置条件

二戒：必选步骤遗漏
   → 必标记：每步标记属性，执行后确认所有必选项已完成

三戒：禁止事项违反
   → 必识别：执行前列出所有禁止事项，操作前检查是否触犯

四戒：输出标准不达标
   → 必验证：逐项对照规范要求，确认输出包含所有必需内容

五戒：前置条件忽略
   → 必检查：逐项验证前置条件，全部满足才开始执行

六戒：执行中对照缺失
   → 必对照：关键节点暂停，对照规范确认执行正确

七戒：验证环节敷衍
   → 必认真：逐项验证，不凭感觉，不走过场
```

### 遵循自检十问

```
执行前：
1. 我是否完整阅读了规范？
2. 我是否识别了所有必选步骤？
3. 我是否识别了所有禁止事项？
4. 我是否明确了输出标准？
5. 我是否验证了前置条件？

执行中：
6. 我是否在关键节点对照了规范？
7. 我是否及时发现并修正了偏差？

执行后：
8. 我是否逐项验证了输出完整性？
9. 我是否验证了输出符合规范标准？
10. 我是否确认可以输出结果？
```

### 遵循失败信号识别

```
出现以下信号，立即警觉：

⚠️ "规范太长，我只读了核心部分"
⚠️ "这个步骤应该是可选的"
⚠️ "禁止事项应该不适用于这个场景"
⚠️ "输出看起来差不多了"
⚠️ "验证一下就行，不用太细"
⚠️ "时间紧，先输出再说"

发现信号 → 立即停止 → 重新审视 → 确保遵循
```
