## 概述

执行质量是 AI Agent 完成任务的核心能力。本策略提供系统化的质量保障方法，确保 AI Agent 高质量完成任务。

**适用场景**：
- 需要提高任务执行成功率
- 需要确保输出质量达标
- 需要优化执行效率

**核心原则**：
- 需求优先：准确理解用户真实需求
- 方法得当：选择最适合的执行方法
- 质量为本：多维度验证输出质量

## 核心策略框架

```
执行质量 = 需求理解 × 方法选择 × 质量验证
```

| 维度 | 核心目标 | 关键动作 |
|-----|---------|---------|
| 需求理解 | 准确把握目标 | 识别意图、明确标准、识别约束 |
| 方法选择 | 选对执行路径 | 评估方法、规划步骤、预估资源 |
| 质量验证 | 确保输出达标 | 完整性验证、准确性验证、可用性验证 |

## 策略一：需求理解

**核心目标**：准确理解用户真实需求，避免理解偏差

### 1.1 意图识别

**识别用户的真实意图**：

```
【表层意图】
用户明确表达的需求
- 用户说了什么？
- 用户要求做什么？

【深层意图】
用户未明确表达但期望的目标
- 用户为什么需要这个？
- 用户希望解决什么问题？
- 用户期望达到什么效果？

【隐含需求】
用户未表达但合理的需求
- 基于场景推断的合理需求
- 基于常识判断的必要需求
```

**意图识别方法**：
```
1. 分析用户表达的显性需求
2. 推断用户的隐性目标
3. 识别任务的上下文环境
4. 确认用户期望的成果形态
```

### 1.2 成功标准明确

**明确任务完成的成功标准**：

```
【硬性标准】
必须达到的最低要求
- 输出必须包含什么？
- 输出必须达到什么质量？
- 输出必须在什么时间内完成？

【软性标准】
期望达到的优化目标
- 输出最好是怎样的？
- 输出最好具备哪些特点？
- 输出最好如何呈现？

【验证标准】
判断是否成功的标准
- 如何判断任务完成？
- 如何判断输出合格？
- 如何判断用户满意？
```

**标准明确模板**：
```
任务：[任务名称]

成功标准：
1. 必须达到：[硬性标准]
2. 期望达到：[软性标准]
3. 如何验证：[验证标准]

失败标准：
1. 缺少必需项：[具体内容]
2. 质量不达标：[具体标准]
3. 超出约束：[具体约束]
```

### 1.3 约束识别

**识别任务执行的约束条件**：

```
【资源约束】
- 可用资源有哪些？
- 资源限制是什么？
- 资源是否充足？

【时间约束】
- 截止时间是什么？
- 时间是否充裕？
- 是否需要分阶段完成？

【质量约束】
- 质量底线是什么？
- 是否允许错误？
- 准确性要求多高？

【范围约束】
- 任务边界在哪里？
- 哪些是必须做的？
- 哪些是不要做的？
```

### 1.4 需求理解验证

**自我验证理解是否正确**：

```
【需求确认】
□ 我能用一句话说明用户要什么吗？
□ 我知道用户为什么要这个吗？
□ 我知道成功标准是什么吗？
□ 我知道约束条件有哪些吗？

【不确定性识别】
□ 是否有模糊的需求？
□ 是否有不明确的约束？
□ 是否有多种可能的理解？

【澄清决策】
□ 需求明确 → 开始执行
□ 存在模糊 → 主动询问用户澄清
□ 可能误解 → 向用户确认理解
```

## 策略三：质量验证

**核心目标**：多维度验证输出质量

### 3.1 完整性验证

**验证输出是否完整**：

```
【内容完整性】
□ 是否包含所有必需内容？
□ 是否有遗漏的部分？
□ 是否有未完成的项？

【结构完整性】
□ 结构是否完整？
□ 各部分是否齐全？
□ 逻辑是否连贯？

【信息完整性】
□ 信息是否充分？
□ 是否有缺失的信息？
□ 是否需要补充？
```

### 3.2 准确性验证

**验证输出是否准确**：

```
【数据准确性】
□ 数据是否正确？
□ 计算是否准确？
□ 引用是否正确？

【逻辑准确性】
□ 推理是否正确？
□ 结论是否合理？
□ 逻辑是否自洽？

【表达准确性】
□ 表达是否准确？
□ 是否存在歧义？
□ 是否容易误解？
```

### 3.3 可用性验证

**验证输出是否可用**：

```
【功能可用性】
□ 输出是否能满足用户需求？
□ 输出是否能解决用户问题？
□ 输出是否达到预期效果？

【易用性】
□ 输出是否易于理解？
□ 输出是否易于使用？
□ 输出是否易于维护？

【兼容性】
□ 输出是否符合用户环境？
□ 输出是否与现有系统兼容？
□ 输出是否易于集成？
```

### 3.4 质量验证流程

```
【第一轮：完整性验证】
检查输出是否包含所有必需内容
→ 不完整：补充缺失内容
→ 完整：进入下一轮验证

【第二轮：准确性验证】
检查输出内容是否准确
→ 不准确：修正错误内容
→ 准确：进入下一轮验证

【第三轮：可用性验证】
检查输出是否满足用户需求
→ 不可用：优化输出
→ 可用：确认输出

【最终决策】
□ 通过所有验证 → 输出结果
□ 存在问题 → 修正后重新验证
```

### 执行质量自检清单

```
【需求理解阶段】
□ 我是否理解了用户的真实意图？
□ 我是否明确了成功标准？
□ 我是否识别了所有约束？
□ 我是否验证了理解正确性？

【方法选择阶段】
□ 我是否评估了多种方法？
□ 我是否选择了最适合的方法？
□ 我是否规划了执行步骤？
□ 我是否预判了可能风险？

【质量验证阶段】
□ 我是否验证了输出完整性？
□ 我是否验证了输出准确性？
□ 我是否验证了输出可用性？
□ 我是否进行了多轮验证？
```

### 问题诊断矩阵

| 问题现象 | 可能原因 | 自我检查点 |
|---------|---------|-----------|
| 输出不符合需求 | 需求理解偏差 | 重新确认用户意图 |
| 执行效率低 | 方法选择不当 | 重新评估方法适用性 |
| 输出质量差 | 验证不充分 | 加强质量验证环节 |
| 需要多次返工 | 质量把控不足 | 增加检查点密度 |

## 应用示例

### 示例1：确保需求理解准确

**场景**：用户请求生成报告

**执行过程**：
1. **意图识别**：
   - 表层：生成一份报告
   - 深层：用户需要用报告做决策
   - 隐含：报告需要简洁、数据准确

2. **标准明确**：
   - 硬性：包含3个核心指标、格式规范
   - 软性：简洁、易读、可视化
   - 验证：能否支持用户决策

3. **约束识别**：
   - 资源：有历史数据
   - 时间：今天内完成
   - 质量：数据准确

4. **验证理解**：
   - 向用户确认："您需要的报告是用于决策，对吗？"
   - 确认后再执行

### 示例2：选择最优执行方法

**场景**：处理大量数据

**执行过程**：
1. **方法识别**：
   - 方法A：逐条处理，准确性高但速度慢
   - 方法B：批量处理，速度快但可能遗漏

2. **方法对比**：
   - 方法A：准确性高、耗时长、适合关键数据
   - 方法B：效率高、风险中等、适合一般数据

3. **方法选择**：
   - 当前场景：数据量大但有质量要求
   - 选择：方法B + 抽样验证
   - 原因：兼顾效率和准确性

### 示例3：多轮质量验证

**场景**：生成代码文件

**执行过程**：
1. **第一轮验证（完整性）**：
   - 检查是否包含所有必需函数 → 通过
   - 检查是否包含必要注释 → 补充后通过

2. **第二轮验证（准确性）**：
   - 检查语法是否正确 → 发现错误 → 修正
   - 检查逻辑是否正确 → 通过

3. **第三轮验证（可用性）**：
   - 检查是否满足需求 → 通过
   - 检查是否易于理解 → 优化变量名 → 通过

4. **最终决策**：输出代码文件
